# 输出格式

每次调查结果都使用以下结构。

## 必需章节

### 1. 接口信息

包含：
- API 路径
- HTTP 方法
- 接口名称（如果可获得）
- 现象摘要

### 2. 日志证据

包含：
- 日志文件路径
- 匹配到的关键词
- 相关的简短片段
- 如果未找到直接日志证据，需要说明

### 3. 入口定位

包含：
- 路由文件
- 控制器（Controller）/ 视图（View）/ 处理器（Handler）位置
- 如有必要，给出框架判断
- 如果无法唯一定位，给出 Top 3 入口候选

### 4. 调用链候选

包含：
- 路由（Route）到控制器（Controller）/视图（View）
- 控制器（Controller）/视图（View）到服务（Service）
- 服务（Service）到模型（Model）/仓储（Repository）/客户端（Client）
- 如果仍有歧义，给出 Top 3 调用链候选

### 5. 最可能原因（Top 1 到 Top 3）

对于每个原因，包含：
- 原因标题
- 原因摘要
- 精确代码位置
- 为什么它很可能成立

### 6. 问题代码位置

列出所有与诊断直接相关的关键文件与符号。

### 7. 判断依据

使用明确证据解释判断基础：
- 路由匹配
- 日志信息
- 校验分支
- 权限分支
- 查询分支
- 外部依赖标记
- 配置读取

### 8. 验证步骤

仅提供只读验证。

允许示例：
- 检查附近的路由声明
- 检查序列化器（Serializer）或模式（Schema）字段
- 检查鉴权或权限分支
- 检查 ORM 过滤条件
- 检查外部客户端错误处理
- 将时间戳与日志片段进行对照

禁止示例：
- 修改代码
- 修改配置
- 重跑迁移
- 写入数据库
- 清理缓存

### 9. 结论

总结：
- 当前最强结论
- 仍然不确定的部分
- 下一步应在只读模式下优先验证什么

## 质量规则

- 每个原因都必须有代码位置。
- 不要在没有证据的情况下使用诸如 `probably a bug in business logic` 之类的模糊表述。
- 如果信息不足，要明确说明。
- 如果路由或入口存在歧义，输出 Top 3 候选。
- 如果没有找到日志，要写明 `No direct log evidence found`，并依赖代码证据。
- 不要包含修复补丁、实现步骤或配置修改。
